# 第3课：ROP 攻击 —— 黑客的"积木"游戏

> **要解决的问题**：不注入任何新代码，黑客怎么劫持你的程序？
> **学完能干什么**：理解 ROP 的攻击机理、gadget 的概念、以及为什么分支密度是检测它的天然特征。

---

## 3.1 正常的函数调用是怎么工作的？

```c
int main() {
    int x = add(3, 4);   // 调用 add
    return x;             // add 返回后从这里继续
}
```

编译成汇编后：

```
地址 0x100:  call 0x200       →  把 0x104（返回地址）压栈，跳到 add
地址 0x104:  ...              →  add 返回后回到这里
...
地址 0x200:  add a, b          →  函数体
地址 0x204:  ret               →  从栈上弹出返回地址 = 0x104，跳回去
```

**栈在调用 add 之后长这样**：

```
栈顶 →  [0x104]  ← 返回地址（call 指令自动压进去的）
        [a=3]     ← add 的参数
        [b=4]
```

`ret` 指令不关心栈上的地址是谁放的、是不是真的返回地址。它只是机械地：弹出栈顶的值 → 跳过去。**这正是漏洞所在。**

## 3.2 栈溢出：改写返回地址

如果一个函数里用了 `strcpy(buf, input)` 且没有长度检查：

```c
void vulnerable(char* input) {
    char buf[10];
    strcpy(buf, input);  // ← 如果 input 超过 10 字节...
}
```

**内存布局**：

```
[buf[0..9]] [编译器放的填充] [返回地址]
                              ↑
                         超长的 input 覆盖了这里
```

攻击者精心构造 input，让返回地址变成他想去的任何地方。

---

## 3.3 第一代攻击：注入 Shellcode（90 年代）

攻击者在 input 里直接放机器码（shellcode），把返回地址指向 shellcode 在栈上的位置。`ret` 跳过去执行 → 拿到 shell。

**防御（2004 年）**：DEP / W⊕X — 内存页要么可写，要么可执行，不能同时。栈可写但不可执行 → shellcode 跑不起来。

---

## 3.4 第二代攻击：ROP（2007 年）—— 不注入，只复用

既然不能执行攻击者的代码，那就**复用已有的合法代码**。程序里本来就有可执行的指令，攻击者只需要控制执行顺序。

**这就是 ROP（Return-Oriented Programming，返回导向编程）**。

---

## 3.5 Gadget：ROP 的"积木块"

Gadget 是程序中本来就存在的、以 `ret` 结尾的短指令序列。

比如程序某处有这段编译出来的代码：

```
地址 0x500:  pop rdi          ← 从栈上弹一个值到 rdi（x10）
地址 0x501:  ret              ← 返回到栈上的下一个地址
```

正常情况下，这是某个函数的一部分。但在攻击者眼里，这就是一个 **Gadget**：先设置 rdi 的值，然后跳到下一个 gadget。

攻击者把栈布置成这样：

```
栈顶 →  [0x500]   ← ret 弹出这个地址，跳过去
        [0x1234]  ← 这个值会被 pop rdi 吃掉（放进 rdi = system() 的参数）
        [0x600]   ← 第一个 gadget 的 ret 弹出这个，跳过去
        [0x5678]  ← 第二个 gadget 的参数
        [0x700]   ← 继续串联
        ...
```

**CPU 执行流程**：

```
1. ret     → 弹出 0x500，跳到 0x500
2. pop rdi → 弹出 0x1234，rdi = 0x1234
3. ret     → 弹出 0x600，跳到下一个 gadget
4. ...     → 持续串联，直到达成攻击目标（比如 system("/bin/sh")）
```

## 3.6 为什么 ROP 会产生密集的分支？

**正常程序**（以 Dhrystone 为例）：

```
条件分支(if/for) → 很多计算指令 → 很久以后 → 函数调用
函数调用 → 很多计算指令 → 很久以后 → 函数返回
分支密度：~1/24 周期
```

**ROP 攻击**：

```
gadget 1: 3条指令 → ret (分支!)
gadget 2: 4条指令 → ret (分支!)
gadget 3: 2条指令 → ret (分支!)
gadget 4: 5条指令 → ret (分支!)
分支密度：~1/3-10 周期（取决于 CPU 的 CPI）
```

**关键差异**：正常程序的分支之间有很多**真正的计算**。ROP 的 gadget 之间几乎没有——gadget 只是"设置参数 + 跳转 + 设置参数 + 跳转"。**所以 ROP 的 branch 密度天然比正常程序高 3-6 倍。**

---

## 3.7 为什么"只听 RET"是作弊级优化？

ROP 攻击的每个 gadget 都以 **RET** 结尾。但正常程序的 branch_taken 信号包含三种指令：

| 指令类型 | 正常密度 | ROP 密度 | 说明 |
|----------|:---:|:---:|------|
| BEQ/BNE（条件分支） | ~1/24 | ~0 | for/while 循环 |
| JAL（函数调用） | ~1/100 | ~0 | 函数 call |
| **JALR/RET**（函数返回） | **~1/150** | **~1/10.8** | **ROP gadget 的链节！** |

**如果只看 RET**：正常程序 ~1/150 周期 v.s. ROP ~1/10.8 周期 → **14 倍差距，随便分**。

**如果看所有分支**：正常程序 ~1/24 周期 v.s. ROP ~1/10.8 周期 → **2 倍差距，神仙也分不出来**。
