# 第2课：PicoRV32 内部解剖 —— 多周期 CPU 的"心跳"

> **要解决的问题**：PicoRV32 怎么执行指令？为什么知道它的 CPI 对安全检测至关重要？
> **学完能干什么**：读懂 PicoRV32 的状态机，知道每类指令消耗几个周期，理解"CPI 修正"的必要性。

---

## 2.1 先搞清楚：PicoRV32 不是"流水线"CPU

很多教材讲 CPU 都是五级流水线。但 PicoRV32 不是。

**流水线 CPU**：同时有 5 条指令在 CPU 里，就像洗衣房的 5 个人同时在干活。吞吐量高（1 CPI），但硬件复杂。

**多周期 CPU**（PicoRV32）：同一时刻只有 1 条指令在 CPU 里。执行完一条才开始下一条。硬件简单（750-2000 LUT），但每条指令需要 3-5 个周期。

**为什么这个区别对我们的安全方案至关重要？**

ROP 攻击在流水线 CPU 上每个周期都能产生一次分支（CPI=1），在 PicoRV32 上每 3-4 个周期才能产生一次（CPI=3-4）。如果我们的神经元参数是按 CPI=1 调的，换到 CPI=3 的 CPU 上就彻底失效了——这就是原论文翻车的根本原因。

---

## 2.2 PicoRV32 的状态机：CPU 的"心跳"

PicoRV32 用 8 个状态（one-hot 编码）来描述 CPU 在任意时刻"在做什么"：

```
状态            含义
cpu_state_trap   = 8'b10000000   出错了，停机
cpu_state_fetch  = 8'b01000000   正在取指令（从内存读）
cpu_state_ld_rs1 = 8'b00100000   正在读第一个源寄存器
cpu_state_ld_rs2 = 8'b00010000   正在读第二个源寄存器
cpu_state_exec   = 8'b00001000   正在执行运算/跳转
cpu_state_shift  = 8'b00000100   正在做移位操作（多周期移位器用）
cpu_state_stmem  = 8'b00000010   正在写内存（store 指令用）
cpu_state_ldmem  = 8'b00000001   正在读内存（load 指令用）
```

**为什么用 one-hot 编码（8 位）而不是 binary（3 位就够了）？**

这是 FPGA 优化的标准做法。one-hot 让状态判断变成"检查某一位是不是 1"——只需要一个 LUT 输入。binary 需要 3 位比较器。代价是多用 5 个 FF，但 FPGA 上 FF 便宜，LUT 贵。

## 2.3 不同指令走过不同的状态序列

### add 指令（寄存器运算）：3 个周期

```
fetch ──→ ld_rs1 ──→ exec ──→ fetch(下一条)
  │          │          │
  读指令    读x5,x6    x5+x6→x10
```

### lw 指令（从内存读取）：4-5 个周期

```
fetch ──→ ld_rs1 ──→ exec ──→ ldmem ──→ fetch(下一条)
  │          │          │          │
  读指令    读基地址   算地址    读内存数据→寄存器
```

### sw 指令（写入内存）：4 个周期

```
fetch ──→ ld_rs1 ──→ ld_rs2 ──→ stmem ──→ fetch(下一条)
  │          │          │           │
  读指令    读基地址   读要存的值   写入内存
```

### JAL（直接跳转）：1 个周期！

```
fetch ──→ fetch(下一条，在跳转目标地址)
  │
  译码出JAL，马上算目标地址，下周期直接去取新指令
```

**JAL 是唯一能在取指阶段就完成的分支**。因为跳转目标在指令里直接写死了（立即数形式）。

### JALR / RET：3 个周期

```
fetch ──→ ld_rs1 ──→ exec ──→ fetch(下一条)
  │          │          │
  读指令    读x1(ra)   跳转到x1存的地址
```

### BEQ（条件分支）：3 个周期

```
fetch ──→ ld_rs1 ──→ exec ──→ fetch(下一条)
  │          │          │
  读指令    读两个操作数  比较，相等则跳转
```

## 2.4 CPI 总结表：这是修正原论文 bug 的关键数据

| 指令类型 | CPI（每个指令需要几个周期） | 对应的高层概念 |
|----------|:---:|------|
| JAL | **1** | 直接函数调用 |
| JALR/RET | **3** | 函数返回 / ROP gadget 的"链节" |
| BEQ/BNE 等条件分支 | **3** | if/for/while 的判断 |
| add/sub/and/or | **3** | 普通计算 |
| lw（读内存） | **4-5** | 读取变量 |
| sw（写内存） | **4** | 写入变量 |

## 2.5 修正原论文的 bug：CPI=1 的灾难

原论文的 testbench 把 gadget 的指令个数直接当成了**周期数**：

```verilog
// 原论文的 bug：gadget_len = 指令数，不是周期数
rop_gadget_len[0] = 3;  // 3条指令的gadget → 3个周期就产生下一个RET
rop_gadget_len[1] = 4;  // 4条指令的gadget → 4个周期
```

但实际上，PicoRV32 需要 **3 个周期才执行一条 RET**，而 gadget 里的其他指令（通常 1-5 条）每条也需要 3-4 个周期。

**正确计算**：一个 3 条指令的 gadget（含最后的 RET），实际消耗约 9-12 个周期。

**影响**：原论文声称的 6.7 倍密度比（ROP vs 正常）是用 CPI=1 算出来的。修正到 CPI=3 后，密度比掉到 2.2 倍——单神经元无论如何都分不出来。

**我们修正后的 testbench**：

```verilog
// 修正：gadget_len = 指令数 × 3 (CPI)
rop_gadget_len[0] = 9;   // 3条指令 × 3 CPI = 9 周期
rop_gadget_len[1] = 12;  // 4条指令 × 3 CPI = 12 周期
```
