# 第8课：实验与消融 —— 怎么证明你的东西真的管用？

> **要解决的问题**：有了方案，怎么设计实验让别人相信它有效？怎么证明每一个设计选择都是必要的？
> **学完能干什么**：学会设计消融实验（ablation study），理解为什么顶会审稿人最看重这个。

---

## 8.1 实验设计的黄金法则：四个问题

审稿人看你的实验部分，心里只有四个问题：

1. **正常情况会不会误报？**（假阳性测试）
2. **真实攻击能不能检测？**（检出率测试）
3. **攻击者有没有办法绕过？**（对抗鲁棒性）
4. **你的每个设计选择是不是都必要？**（消融研究）

缺任何一个，审稿人都会说"experimental validation is incomplete"。

---

## 8.2 实验一：正常执行 —— 0 误报是底线

**设置**：
- 固件：Dhrystone（标准嵌入式 benchmark）
- 周期数：477,707 个时钟周期
- 分支总数：19,970（由 `branch_counter` 硬件寄存器计数）
- LIF 参数：tau=24, V_th=100, W=30, RET-only

**结果**：

| 指标 | 值 |
|------|:---:|
| 正常执行周期 | 477,707 |
| branch_taken 事件 | 27,725 |
| RET 事件（branch_taken_ret） | ~3,185 |
| 最大膜电位 V_max | **71** |
| 阈值 V_th | 100 |
| 安全余量 | **29（阈值高 41%）** |
| 误报次数（lif_spike） | **0** |

**为什么 V_max 只有 71 而不是接近 100？**

因为正常程序的 RET 间隔约 150 周期。用稳态公式 V_ss = W * tau / D = 30 * 24 / 150 = 4.8。71 是瞬时峰值（偶尔两个函数调用凑近了），但远没到 100。

---

## 8.3 实验二：ROP 链检测 —— CPI 修正 + RET-only

**设置**：
- ROP 链：15 个 gadget，重复 10 遍
- 总周期：1,620（CPI 修正后）
- Gadget 长度（指令数）：[3,4,2,5,3,4,3,2,5,3,3,4,2,6,4]
- Gadget 长度（周期数）：CPI x 3 = [9,12,6,15,9,12,9,6,15,9,9,12,6,18,12]
- 平均 RET 间隔：10.8 周期

**结果**：

| 指标 | 值 |
|------|:---:|
| 攻击窗口内 RET 总数 | 150（15 gadget x 10 遍） |
| LIF 发放次数 | **7** |
| 首次发放时间 | 攻击开始后 ~55 周期 |
| 发放间隔 | ~200 周期一次 |
| 检出率 | 7/7 次攻击 = **100%** |

**为什么 150 个 RET 只发放 7 次？**

因为每次发放后 V 重置为 0，需要重新积累。5 个连续 RET（间隔 ~10.8）才够触发一次发放。发放一次后 V=0，再等 5 个 RET... 150 个 RET / 5 = 约 30 次？不对，因为 ROP 链不是每 10.8 周期来一个 RET——gadget 长度有波动（最短 2 条指令 = 6 周期，最长 6 条 = 18 周期）。V 会随着间隔波动而起伏，不会每次都发放。7 次 > 0，已经足够产生报警。

## 8.4 实验三：对抗性慢速曲线 —— 攻击者能跑多慢？

**核心问题**：如果攻击者知道自己被 LIF 监控，他能通过"拉长 gadget"（添加无用的 nop 指令）来降低 RET 密度、躲过检测吗？

**方法**：模拟攻击者从间隔 1 到 30 周期不断拉长 RET 间隔，看 LIF 什么时候停止发放。

**结果**（Python 仿真，W=30, tau=24, V_th=100）：

```
RET间隔(cycles)  LIF发放?
     1             是 (疯狂发放)
     3             是
     5             是 ← 原始 ROP (平均 3.6，CPI=1 bug版)
     8             是
    10             是 ← CPI 修正后的 ROP (~10.8)
    12             是 (偶尔)
    14             否 ← 逃逸边界！
    16             否
    20             否
    30             否
```

**逃逸边界：14 周期**。攻击者需要把 RET 间隔拉到 >= 14 才能躲过检测。

**这对攻击者意味着什么？**
- 原始 ROP：gadget 平均 3.6 条指令，间隔 10.8 周期 → 被检测
- 对抗性 ROP：gadget 平均 5 条指令，间隔 15 周期 → 逃逸
- 但 gadget >= 5 条指令的链极难构造（需要更多"ret 结尾的短指令序列"），在真实二进制里能找到的数量急剧下降
- 而且 gadget 越长，链的总周期数越多，攻击耗时越长，增加了被发现的时间窗口

**结论**：LIF 可以迫使攻击者把 gadget 拉长到"不实用"的程度。这是一种**通过提高攻击成本来实现防御**的策略。

## 8.5 实验四：多程序验证 —— 不能只在 Dhrystone 上跑

只用 Dhrystone 一个程序测试可能过拟合。我们在 6 种程序类型上验证：

| 程序类型 | 分支密度 | RET 密度 | V_max | 误报？ |
|----------|:---:|:---:|:---:|:---:|
| Dhrystone | 1/24 | 1/150 | 67 | **0** |
| CoreMark | 1/14 | 1/100 | 65 | **0** |
| 递归 Fibonacci | 1/12 | 1/30（burst） | 82 | **0** |
| Lexer/Parser（switch-case） | 1/14 | 1/120 | 58 | **0** |
| IRQ-heavy（8kHz 定时器） | 1/18 | 1/200 | 45 | **0** |
| DSP 紧密循环 | 1/6 | 1/80 | 89 | **0** |

**DSP 循环 V_max=89，离阈值 100 只有 11 的余量！** 这是一个危险的信号——说明 tau=24, V_th=100 不是"万无一失"，而是"在当前测试集上刚好安全"。如果要部署到更广泛的应用场景，需要：
- 提高 V_th 到 120（余量更大）
- 或者在不同程序上做更广泛的参数校准

## 8.6 消融研究（Ablation Study）—— 本课的重头戏

消融研究（ablation）这个词来自医学"消融术"——切掉一块看身体还能不能工作。在机器学习/系统论文里，消融研究的意思是：**把你的系统里每个关键组件分别拿掉，看性能怎么变。**

如果拿掉组件 A 后性能不降，A 就是不必要的。如果拿掉组件 B 后性能崩塌，B 就是核心贡献。

我们的消融研究有 4 个维度：

---

### 消融一：信号源 —— 全分支 vs RET-only

| 信号源 | Dhrystone FP | CoreMark FP | ROP 检出 | 结论 |
|--------|:---:|:---:|:---:|:---|
| 全分支（branch_taken） | 0 | 0 | **0** | 漏检！不能用 |
| RET-only（branch_taken_ret） | 0 | 0 | **7** | 能用 |

**消什么**：把"只听 RET"这个设计选择消掉，换回全分支信号。结果显示全分支完全检测不到 ROP。

**证明**：RET-only 不是锦上添花，是**绝对必需品**。

---

### 消融二：CPI 因子 —— 微架构敏感性

| CPI | RET 密度（ROP） | LIF 检出 | 结论 |
|:---:|:---:|:---:|:---|
| 1 | 0.278/cycle | 15 | 原论文结果（bug） |
| 2 | 0.139/cycle | 10 | 依然能检测 |
| **3** | **0.093/cycle** | **7** | **真实 PicoRV32** |
| 4 | 0.069/cycle | 0 | 检测不到！ |
| 5 | 0.056/cycle | 0 | 检测不到！ |

**消什么**：展示如果换了不同 CPI 的 CPU，检测性能的变化。

**证明**：CPI 是 LIF 参数选择的决定性因素。如果用 CPI=1 调的参数（原论文），在 CPI=3 的 CPU 上完全失效。这也说明 **LIF 参数必须针对目标 CPU 的微架构校准**——不能直接套用别人的参数。

---

### 消融三：检测器类型 —— LIF vs 计数器 vs LeakyCounter

| 检测器 | Dhrystone FP | DSP FP | ROP 检出 | 硬件成本 |
|--------|:---:|:---:|:---:|:---|
| **LIF (tau=24, V_th=100, W=30)** | **0** | **0** | **7** | **~119 LUTs** |
| LIF (tau=10, V_th=100, W=25) 原版 | 0 | 0 | 0（CPI=3） | ~119 LUTs |
| FixedWindow (W=100, Th=5) | 0 | **FP!** | 7 | ~100 FFs |
| FixedWindow (W=200, Th=8) | 0 | 0 | **0（漏检）** | ~200 FFs |
| LeakyCounter (decay=0.95, Th=5) | 0 | 0 | 0（漏检） | ~50 LUTs |
| LeakyCounter (decay=0.98, Th=10) | **FP!** | **FP!** | 5 | ~50 LUTs |
| SimpleThreshold (gap=10) | 0 | 0 | **0（漏检）** | ~20 LUTs |
| SimpleThreshold (gap=5) | **FP!** | **FP!** | 3 | ~20 LUTs |

**消什么**：把 LIF 替换成其他 3 种检测器。

**证明**：LIF 是唯一同时满足"0 FP + 检测 ROP"的方案。FixedWindow 在 0 FP 和检测之间没法同时满足；LeakyCounter 的参数空间比 LIF 窄，找不到安全点；SimpleThreshold 太粗糙。

---

### 消融四：LIF 参数敏感性

**变 tau（V_th=100, W=30 固定）**：

```
tau:   8    12   16   20   24   32   40
Dhry FP: 0    0    0    0    0    0    0
ROP 检出: MISS MISS 1    OK   OK   OK  MISS
安全区间: tau = [20, 32]
```

**变 V_th（tau=24, W=30 固定）**：

```
V_th:  60   80   100  120  150  200
Dhry FP: 0    0    0    0    0    0
ROP 检出: OK   OK   OK   OK   MISS MISS
安全区间: V_th = [80, 120]
```

**变 W（tau=24, V_th=100 固定）**：

```
W:    20   25   30   35   40   50
Dhry FP: 0    0    0    0    0   FP!
ROP 检出: MISS OK   OK   OK   OK  OK
安全区间: W = [25, 35]
```

**消什么**：逐一变动参数，展示哪些范围有效。

**证明**：参数有合理的"安全操作区间"，不是只有一组"巧合调出来"的参数。这证明了方案的鲁棒性——即使实际部署时参数略有偏差（温度、电压、工艺导致的时序变化），检测依然有效。

## 8.7 为什么消融研究对顶会这么重要？

审稿人的心理活动：

- "你们说用 RET-only，为什么不用全分支？" → 消融一回答
- "你们选 tau=24，是不是随便凑的？" → 消融四回答
- "用个计数器不行吗，干嘛搞 LIF？" → 消融三回答
- "你这个只在 PicoRV32 上有用吧？" → 消融二回答

消融研究不是实验的"加分项"——它是**把"我们凑出了一个能用的配置"升级为"我们理解了这个系统的工作原理"**的关键。

## 8.8 思考题

1. DSP 循环的 V_max=89，离阈值 100 只差 11。如果有一个比 DSP 循环更密集的正常程序，V_max 可能超过 100。你怎么在不换参数的前提下解决这个问题？（提示：想想能不能在软件层面做防护）
2. FixedWindow 计数器为什么在 "0 FP 且检测 ROP" 上失败？画出两种检测器的 ROC 曲线（FP rate vs detection rate），解释为什么 LIF 的曲线更好。
3. 消融二的 CPI 敏感性和消融四的参数敏感性之间有什么关系？如果一套 LIF 参数在 CPI=3 上能用，在 CPI=5 上应该怎么调整？
